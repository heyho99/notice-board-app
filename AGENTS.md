# プロジェクト指示書

## 1. プロジェクト概要

- 名称: 社内お知らせ掲示板Webアプリ
- 目的: 社内お知らせの投稿・閲覧・編集・削除ができる、学習用の小規模 Web アプリ
- 認証なし・データ非永続化（インメモリ）の最小構成

詳細な要件は `README.md` を参照。

## 2. 技術スタック（厳守）

以下の構成から外れないこと。新しいライブラリやフレームワークの導入は事前に学習者へ確認すること。

- フロントエンド
  - 生の JavaScript のみ（React / Vue / jQuery などのフレームワーク・ライブラリ禁止）
  - HTML / CSS は素の状態で記述
- バックエンド
  - Python 3.11
  - Flask
  - venv による仮想環境
  - インメモリデータ（SQLite / 外部 DB / ファイル永続化はすべて禁止）

## 2.1 コマンド実行時のvenv使用規則

シェルコマンドは実行のたびに別プロセスとして起動するため、`activate` の状態は引き継がれない。
また、PowerShell の実行ポリシー（Restricted 等）により `.ps1` スクリプトはブロックされる場合がある。
そのため、本プロジェクトでは **venv をアクティベートせず、venv 内の実行ファイルをフルパスで直接呼び出す** こと。

### 原則

- venv 配下のコマンドは必ず `venv\Scripts\<コマンド名>` の形式で実行する（`.exe` を直接呼び出す扱いになり、PowerShell の実行ポリシーの影響を受けない）。
- `python` / `pip` / `pytest` 等を **単体で実行してはならない**（グローバル環境が使われるため）。
- `venv\Scripts\activate` / `venv\Scripts\Activate.ps1` / `deactivate` は **使用しない**（PowerShell では実行ポリシーで失敗、cmd でも親プロセスに反映されない）。
- 列挙にないコマンドを使うときは、原則として `venv\Scripts\python -m <モジュール名>` の形式で実行する（最も汎用的で安全）。

### よく使うコマンド

| やること | 使うコマンド |
| --- | --- |
| Flaskサーバ起動 | `venv\Scripts\python app.py` |
| パッケージインストール | `venv\Scripts\pip install <パッケージ名>` |
| パッケージ一覧出力 | `venv\Scripts\pip freeze` |
| requirements.txt から一括導入 | `venv\Scripts\pip install -r requirements.txt` |
| テスト実行（pytest） | `venv\Scripts\pytest` |
| モジュールとして実行（汎用） | `venv\Scripts\python -m <モジュール名>` |

## 3. ドキュメント構成

設計書・仕様書は `README.md` または `docs/` 配下のマークダウンファイルにのみ書く。それ以外の場所には作らない。

事前に定義済みのドキュメント：

| ファイル | 役割 |
| --- | --- |
| `README.md` | 要件書 |
| `docs/data_model.md` | データモデル定義書（変更禁止） |

これら以外の設計書・仕様書のファイル名は事前に固定しない。具体的にどのファイルに書くかは、毎回学習者の指示に従うこと。指示がない場合は、勝手に新規ファイルを作らず学習者に確認する。

`README.md` と `docs/data_model.md` は本プロジェクトの前提であり、勝手に変更しないこと。
変更が必要と判断した場合は、必ず学習者に確認してから行う。

## 4. 開発の進め方

掲示板アプリの機能（例：お知らせの投稿、一覧表示、編集、削除）を 1 つ作るたびに、以下のルールに従う。

- 設計時（仕様書・設計書を作成・更新するとき）: `.claude/rules/design.md`
- 実装時（コードを書くとき）: `.claude/rules/implementation.md`

基本姿勢：

- いま作っている機能の範囲だけを書き・実装する。他の機能まで手を広げない。
- 動作確認は実際にサーバを起動し、ブラウザで触って行う。

成果物を作成したら、次に進む前に学習者に内容を確認してもらうこと。

## 5. 設計・コーディング方針

- 要件・仕様書に書かれた機能だけを、必要最小限の形で実装する。書かれていない機能を勝手に加えない。
- 学習用プロジェクトのため、過剰な抽象化や将来拡張のための仕掛けは入れない。
- コメントは「なぜそうしたか」が非自明な場合のみ最小限。コードの説明的コメントは不要。
- バリデーション・エラーハンドリングは仕様書に書かれた範囲で実装する。仕様外のケースを勝手に増やさない。
- ファイルや関数を新設する前に、既存ファイル内に追加できないか検討する。

詳細な設計時／実装時のルールは `.claude/rules/design.md` と `.claude/rules/implementation.md` を参照。

## 6. やり取りのルール

### 言語

- 応答・コミットメッセージ・コード内コメント・ドキュメントは原則すべて日本語で記述する。
- 専門用語・コマンド・関数名・ライブラリ名など、英語の方が自然なものは無理に訳さない。

### 応答スタイル

- 冗長な前置き・後書きは避け、要点から先に伝える。
- 結果のサマリは短く（1〜3 文程度）。コードの差分そのものは繰り返さない。
- 不明点・要件があいまいな点は、勝手に決めずに学習者へ質問する。

### 確認を取るべきタイミング

次のような場合は、必ず作業前に学習者へ確認する。

- 仕様書（`README.md` / `docs/data_model.md` を含む）を変更する必要があると判断したとき。
- 技術スタックから外れるライブラリ・ツールの導入を提案するとき。
- 一度に複数ファイルにまたがる大きな変更を行うとき。
- 学習者の指示と既存ドキュメントの内容に矛盾があるとき。
- 設計書を、`.claude/rules/design.md` で指定されたファイル以外に書く必要が出たとき。

### やってはいけないこと

- 確認なしに、要件・データモデル・既存仕様を書き換えること。
- 質問されていないのに、関連のないリファクタリングや追加機能を勝手に始めること。
- 出力を英語のみで返すこと（学習者の指示が英語の場合は除く）。

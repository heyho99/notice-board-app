# 社内お知らせ掲示板Webアプリ

社員が業務連絡やお知らせを投稿・閲覧できる、シンプルな社内向け掲示板アプリです。
本リポジトリは、要件書（本ファイル）とデータモデル定義書（`docs/data_model.md`）のみが事前に用意されています。
ここから画面仕様・API仕様・バックエンド・フロントエンドを順番に作り上げていきます。

---

## 1. アプリの目的

- 社内のお知らせを 1 か所に集約し、誰でもブラウザから閲覧できるようにする。
- 投稿・編集・削除をシンプルな操作で行えるようにする。
- 学習用のため、認証や永続化は行わず、最小限の機能に絞る。

## 2. 想定利用シーン

- 総務担当者が、全社向けのお知らせ（重要連絡・メンテナンス情報など）を投稿する。
- 一般社員が、最新のお知らせ一覧をブラウザで確認する。
- 古くなった、または誤って投稿したお知らせを編集・削除する。

## 3. 機能要件

### 3.1 お知らせ一覧

- 登録済みのお知らせを一覧表示できること。
- 新しいお知らせほど上に表示されること（作成日時の降順）。
- 各行に、タイトル・カテゴリ・作成日時を表示すること。
- カテゴリ（`重要` / `一般` / `メンテナンス`）で絞り込みができること。
- お知らせが 1 件もない場合は、その旨をユーザーに分かるよう表示すること。

### 3.2 お知らせ詳細

- 一覧から 1 件選択すると、本文を含む詳細を確認できること。
- 詳細画面から、編集・削除が行えること。

### 3.3 お知らせ投稿（新規作成）

- タイトル・本文・カテゴリを入力して、お知らせを新規投稿できること。
- カテゴリは `重要` / `一般` / `メンテナンス` の 3 種類から選択する。
- タイトル・本文・カテゴリのいずれかが未入力の場合はエラーとし、登録しないこと。
- 投稿後、その内容が一覧に反映されること。

### 3.4 お知らせ編集

- 既存のお知らせのタイトル・本文・カテゴリを編集できること。
- バリデーションは新規投稿と同等とする。
- `id` と `created_at` は変更しないこと。

### 3.5 お知らせ削除

- 既存のお知らせを削除できること。
- 削除前に、ユーザーに確認を求めること。
- 削除後、その項目が一覧から消えること。

## 4. 非機能要件

- 認証・ユーザー管理は行わない。誰でも全機能を利用できる。
- データはアプリ起動中のみ保持できればよい（インメモリ、再起動で消える）。
- 1 名の学習者がローカル PC で動作確認できる規模・性能で十分とする。
- フロントエンドは静的ファイルとして配信し、バックエンドの API を呼び出して動作する。

## 5. 技術スタック

- フロントエンド
  - 生の JavaScript（ライブラリ・フレームワークは使用しない）
  - HTML / CSS
- バックエンド
  - Python 3.11
  - Flask
  - venv（仮想環境）
  - インメモリデータ（外部 DB は使用しない）

## 6. データモデル

詳細は `docs/data_model.md` を参照。
扱うのは `Notice`（お知らせ）モデル 1 種類のみで、ユーザーモデルは扱わない。

## 7. 成果物

最終的に、以下を備えたリポジトリにすること。

- `docs/` 以下に、画面仕様書・API 仕様書・バックエンド仕様書・フロントエンド仕様書が揃っている。
- バックエンド（Flask アプリ）と、それを呼び出すフロントエンド（HTML / CSS / JS）が動作する。
- README に従って、誰でもローカルでアプリを起動し、機能要件を一通り確認できる。

## 8. 進め方（概要）

詳細は教材を参照。大きな流れは次の通り。

1. 要件（本ファイル）とデータモデル（`docs/data_model.md`）を読み込む。
2. 画面仕様書 `docs/screen_spec.md` を作成する。
3. API 仕様書 `docs/api_spec.md` を作成する。
4. バックエンド仕様書 `docs/backend_spec.md` を作成し、それに沿って実装する。
5. フロントエンド仕様書 `docs/frontend_spec.md` を作成し、それに沿って実装する。
6. ローカルで起動し、動作確認・修正を行う。

各ステップでは、Bob（Claude Code）に判断を委ねる部分と、自分で確認・指示する部分を意識して進めること。

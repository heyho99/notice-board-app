# コーディング・実装ルール

本プロジェクトのコードを書くときの共通ルール。学習用なので、シンプル・読みやすさを最優先する。

## 技術スタックの制約

- フロントエンドは生の JavaScript のみ。React / Vue / jQuery などのフレームワーク・ライブラリを導入しない。
- HTML / CSS は素の状態で書く。CSS フレームワーク（Bootstrap, Tailwind 等）は導入しない。
- バックエンドは Python 3.11 + Flask。データはインメモリ。SQLite を含む外部 DB やファイル永続化はしない。
- 仮想環境は `venv/`（プロジェクト直下）に固定。Windows では `.\venv\Scripts\activate` で有効化する。
- `pip install` を行ったら必ず `requirements.txt` を更新する。

## ディレクトリ・ファイル構成

- 同じ責務のコードを複数ファイルに散らさない。1 機能 1 ファイルを意識する。
- 新しいファイルを作る前に、既存ファイルへ追加できないかを検討する。
- 不要になったファイル・コードは残さず削除する（コメントアウトでの「保存」もしない）。

## バックエンド（Flask）

- ルーティング・ビジネスロジック・データアクセスは、最低限の単位で分けて書く（無理に層を増やさない）。
- レスポンスは JSON を基本とする。エラー時は HTTP ステータスとエラーメッセージを返す。
- バリデーションは仕様書に書かれた範囲に限定する。仕様にない検査を勝手に増やさない。
- インメモリデータは、シンプルなリスト・辞書で持つ。ID は単調増加の整数で採番する。
- 例外処理は「壊れたときに 500 を返す」程度に留め、try/except で握りつぶさない。

## フロントエンド（生 JS）

- 1 つの HTML（SPA 風）でも、複数 HTML（画面ごとに分ける）でも構わないが、仕様書の方針に従うこと。
- DOM 操作は `document.querySelector` 等の標準 API を使う。
- API 呼び出しは `fetch` を使う。共通処理（ベース URL、エラー処理）はまとめる。
- ユーザー入力のバリデーションは、まずクライアント側で素早く返し、最終チェックはサーバ側に任せる。
- グローバル変数の乱用を避け、関数 or モジュールで scope を切る。

## コメント・命名

- コメントは「なぜそうしているか」が非自明なときだけ書く。動作の説明コメントは原則書かない。
- 関数・変数名は日本語ローマ字混じりではなく、英語の自然な名前を使う（例: `notices`, `createNotice`）。
- TODO / FIXME を残す場合は、何を、いつ、誰が直すのかが分かる粒度で書く。

## エラー・ログ

- バックエンドのデバッグログは、開発時のみ最低限。本番運用想定の凝ったロガー設定は導入しない。
- フロントエンドの `console.log` は、原因調査が終わったら削除する。

## やってはいけないこと

- 仕様にない機能（ログイン、ファイル添付、ページネーション等）を独断で追加する。
- データ永続化（ファイル書き出し / DB / localStorage 等）を勝手に追加する。
- フレームワーク・ライブラリ・CDN を独断で追加する。
- 既存ドキュメント（要件・データモデル）と矛盾するコードを書く（迷ったらドキュメントを優先し、必要なら学習者に確認する）。

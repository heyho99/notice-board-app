# 開発フロー・作業ルール

本プロジェクトは「仕様 → 実装」を 1 ステップずつ進める学習用プロジェクト。
作業の順序と粒度を守ること。

## 大きな流れ

1. 要件確認（`README.md`）
2. データモデル確認（`docs/data_model.md`）
3. 画面仕様書（`docs/screen_spec.md`）作成
4. API 仕様書（`docs/api_spec.md`）作成
5. バックエンド仕様書（`docs/backend_spec.md`）作成
6. バックエンド実装
7. フロントエンド仕様書（`docs/frontend_spec.md`）作成
8. フロントエンド実装
9. 起動・動作確認・修正

## 各ステップでの守るべき点

- 上のステップで作った成果物（要件・データモデル・各仕様書）を、下のステップが矛盾なく踏襲すること。
- 仕様書を書かずに実装に進まない。仕様の不足に気づいた場合は、まず仕様書を更新する。
- 1 ステップ完了したら、次に進む前に学習者に確認してもらう前提で動く。

## 仕様書の書き方

- データモデル（`docs/data_model.md`）に書かれているフィールドや制約と矛盾しない内容にすること。
- 画面仕様には、画面ごとに「目的」「表示項目」「ユーザー操作」「他画面への遷移」を含めること。
- API 仕様には、エンドポイントごとに「メソッド」「パス」「リクエスト」「レスポンス」「エラー」を含めること。
- バックエンド・フロントエンド仕様には、ファイル構成と、各処理の責務（何をどこで担うか）を含めること。

## 実装の進め方

- バックエンド・フロントエンドとも、まず最小構成（1 機能だけ動く状態）を通してから、機能を足していく。
- 実装中に仕様書の不足・誤りに気づいたら、コードだけで辻褄を合わせず、仕様書側を直してから実装を直す。
- 動作確認は、実際にサーバを起動して画面から行う。仕様書の chart や README だけで「できたつもり」にしない。

## やってはいけないこと

- 一度にすべての画面・API・実装を作ってしまうこと。
- 仕様外の機能（ログイン、ファイル添付、永続化など）を独断で追加すること。
- 複数の仕様書をまとめて変更し、どこを根拠に直したか分からなくすること。

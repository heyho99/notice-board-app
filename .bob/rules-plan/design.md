# 設計時のルール

設計書・仕様書をマークダウンで作成・更新するときに従うルール。

## 機能を作るときの進め方

1. `README.md` と `docs/data_model.md` を読み、これから作る機能の要件と扱うデータを確認する。
2. その機能に必要な範囲だけ、仕様書を作成・更新する（書く場所は下記「設計書を書く場所」を参照）。
3. 仕様書をもとに実装を進める。
4. 実装中に仕様書の不足・誤りに気づいたら、コードだけで辻褄を合わせず、仕様書側を直してから実装を直す。

## 設計書を書く場所

設計書・仕様書は、`docs/` 配下のマークダウンファイルに作成する

具体的にどのファイルに書くかは、毎回学習者の指示に従うこと。指示がない場合・どこに書くべきか迷った場合は、**勝手に新しいファイルを作らず、学習者にファイル名を確認すること**。

- 作成するドキュメントのファイル名が指定されていない場合は、勝手に推測せず、ユーザーに確認する

## 守るべき原則

- 仕様書を書かずに実装に進まない。仕様の不足に気づいたら、まず仕様書を更新する。
- `docs/data_model.md` に書かれているフィールドや制約と矛盾しない内容にする。
- いま作っている機能の範囲だけを書く。他の機能の仕様にまで手を広げない。
- `README.md` と `docs/data_model.md` は本プロジェクトの前提であり、勝手に変更しない。変更が必要と判断したら、必ず学習者に確認してから行う。
- 仕様書を更新したら、次のステップに進む前に学習者に内容を確認してもらう。

## やってはいけないこと

- 学習者に確認せず、勝手に新しい設計書・仕様書ファイルを作ること。
- `docs/` と `README.md` 以外の場所に設計書を書くこと。
- 仕様外の機能（ログイン、ファイル添付、永続化、ページネーション等）を仕様書に追加すること。
- 複数の仕様書をまとめて書き換え、どこを根拠に直したか分からなくすること。

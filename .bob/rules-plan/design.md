# 設計時のルール

設計書・仕様書をマークダウンで作成・更新するときに従うルール。

## 仕様書作成の進め方

1. その機能に必要な範囲だけ、仕様書を作成・更新する。
2. ある要件に対して複数の実装方式が考えられる場合、各方式毎に実装難易度を示したうえで、ユーザーに「どの実装方式を採用するか」を確認する。
   - 各方式の概要
   - 実装難易度(低 / 中 / 高)
   - メリット・デメリット
   - 推奨する方式とその理由

3. 設計や仕様が要件を満たしているかを確認する。

## 仕様書の粒度

- 仕様書は「AIが読んで実装できる」最小限の情報に絞る。
- コードを読めばわかること(変数名、ライブラリの使い方など)は書かない。
- 判断に迷う箇所(命名規則、レイヤー分け、エラーレスポンス形式など)は書く。

## ドキュメント構成

設計書・仕様書は `docs/` 配下のマークダウンファイルにのみ書く。それ以外の場所には作らない。

事前に定義済みのドキュメント：

| ファイル | 役割 |
| --- | --- |
| `docs/requirements.md` | 要件書 |
| `docs/data_model.md` | データモデル定義書（変更禁止） |

これら以外の設計書・仕様書のファイル名は事前に固定しない。具体的にどのファイルに書くかは、毎回ユーザーの指示に従うこと。指示がない場合・どこに書くべきか迷った場合は、**勝手に新しいファイルを作らず、ユーザーにファイル名を確認すること**。

- 作成するドキュメントのファイル名が指定されていない場合は、勝手に推測せず、ユーザーに確認する

## 守るべき原則

- `docs/data_model.md` に書かれているフィールドや制約と矛盾しない内容にする。
- いま作っている機能の範囲だけを書く。他の機能の仕様にまで手を広げない。
- `docs/requirements.md` と `docs/data_model.md` は本プロジェクトの前提であり、勝手に変更しない。変更が必要と判断したら、必ずユーザーに確認してから行う。
- 仕様書を更新したら、次のステップに進む前にユーザーに内容を確認してもらう。

## やってはいけないこと

- ユーザーに確認せず、勝手に新しい設計書・仕様書ファイルを作ること。
- 仕様外の機能（ログイン、ファイル添付、永続化、ページネーション等）を仕様書に追加すること。
- 仕様書に実装コードを長々と貼ること(擬似コードや関数シグネチャ程度に留める)。
- 複数の仕様書をまとめて書き換え、どこを根拠に直したか分からなくすること。

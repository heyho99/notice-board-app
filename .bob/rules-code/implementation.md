# 実装時のルール

コードを書くときの共通ルール。学習用なので、シンプル・読みやすさを最優先する。

## 基本原則

- 要件・仕様書に書かれている機能だけを、必要最小限の形で実装する。書かれていない機能を勝手にリッチにしない。
- 過剰な抽象化や、将来の拡張を見越した仕掛けを入れない。いま必要な分だけ書く。
- 既存ドキュメント（要件・データモデル・各仕様書）と矛盾するコードを書かない。迷ったらドキュメントを優先し、必要なら学習者に確認する。

## 実装時の作業ルール

技術スタック自体の制約は `CLAUDE.md` 2章「技術スタック（厳守）」を参照。実装作業中に守ること：

- 仮想環境は `venv/`（プロジェクト直下）を使う。
  - **venv はアクティベートしない**。venv 内の実行ファイルを `venv\Scripts\<コマンド名>` の形式でフルパス指定して呼び出す（詳細は `AGENTS.md` 2.1 参照）。
  - `venv\Scripts\activate` / `Activate.ps1` / `deactivate` は使用しない（PowerShell の実行ポリシーで失敗、cmd でも親プロセスに反映されないため）。
  - 列挙にないコマンドは `venv\Scripts\python -m <モジュール名>` 形式で実行する。
- `pip install` を行ったら必ず `requirements.txt` を更新する（実行は `venv\Scripts\pip install <パッケージ名>` → `venv\Scripts\pip freeze > requirements.txt`）。
- バックエンドの起動コマンド・フロントエンドの開き方は、実装後に `README.md` 末尾へ追記してよい。

## コメント・命名

- コメントは「なぜそうしているか」が非自明なときだけ書く。動作の説明コメントは原則書かない。
- 関数・変数名は日本語ローマ字混じりではなく、英語の自然な名前を使う（例: `notices`, `createNotice`）。
- TODO / FIXME を残す場合は、何を、いつ、誰が直すのかが分かる粒度で書く。

## エラー・ログ

- バックエンドのデバッグログは、開発時のみ最低限。本番運用想定の凝ったロガー設定は導入しない。
- フロントエンドの `console.log` は、原因調査が終わったら削除する。

## 動作確認

- 動作確認は、実際にサーバを起動して画面から行う。仕様書の chart や README だけで「できたつもり」にしない。
- 実装中に仕様書の不足・誤りに気づいたら、コードだけで辻褄を合わせず、仕様書側を直してから実装を直す。

## やってはいけないこと

- 仕様にない機能（ログイン、ファイル添付、ページネーション等）を独断で追加する。
- データ永続化（ファイル書き出し / DB / localStorage 等）を勝手に追加する。
- フレームワーク・ライブラリ・CDN を独断で追加する。
